# Codex 提示词模板

## 概述

为了不再摇摆，固定 Codex 中提示词的模板。参照：[Codex 最佳实践](https://developers.openai.com/codex/learn/best-practices)

## 固定规范

**精简版**（最常用）：

```markdown
# Goal
[一句话写清最终要完成什么]

# Context
- [相关文件/模块/接口]
- [已知背景]
- [可以使用的工具或数据]

# Constraints
- [不能做什么]
- [必须遵守什么]
- [技术或范围限制]

# Done when
- [完成的判定条件 1]
- [完成的判定条件 2]
- [什么时候该停止]
```

**标准版**：

```markdown
Role: [模型现在扮演什么角色]

# Goal
[一句话写清最终要完成什么]

# Constraints
- [不能做什么]
- [必须遵守什么]
- [技术或范围限制]

# Context
- [相关文件/模块/接口]
- [已知背景]
- [可以使用的工具或数据]

# Output
- [最终回答要包含什么]
- [过程中的沟通方式]
- [格式要求，若有]

# Done when
- [完成的判定条件 1]
- [完成的判定条件 2]
- [什么时候该停止]

# Validation
- [改完后跑什么测试/命令]
- [如果跑不了，要怎么说明]
```

示例：**改代码模板**：

```markdown
Role: 你是当前仓库中的 Python coding agent。

# Goal
实现或修复：[把任务写在这里]

# Constraints
- 不改动无关文件。
- 不覆盖未提交修改。
- 优先小步修改，避免重构。
- 遵循现有代码风格与目录结构。

# Context
- 相关文件：[文件路径]
- 相关行为：[接口/函数/模块]
- 已知问题：[报错/现象/需求]

# Output
- 先说明将检查哪些文件。
- 改完后说明改动内容和原因。
- 最终给出验证结果。

# Done when
- 目标功能已实现或问题已修复。
- 相关代码已更新到仓库中。
- 有最小验证证据支持结果。

# Validation
- 运行：[命令]
- 若无法运行，说明原因和替代检查。
```

## 排障类提示词：先写真实目标，再给复现入口

### 核心原则

不要只给“复现入口”，要先写清楚“真正想解决的问题”。

例如，文件上传类问题里，`/file/upload` 成功通常只是入口，不一定是目标。如果真实问题是“上传后解析失败”，提示词第一段就要明确：

  > 我不关心 `/file/upload` 是否 200，我关心上传后的解析结果是否成功。

这样可以避免 agent 把精力花在上传接口、鉴权、HTTP 状态码上，而跳过真正的业务失败链路。

### 强烈建议写清楚的内容

  1. 真实目标

     明确最终要修什么，而不是只写从哪里开始复现。

     示例：

     > 修复 `.wps` 和空 Excel 上传后解析结果失败的问题。

  2. 非目标

     明确哪些问题不要处理，避免 agent 顺手修无关内容。

     示例：

     > qwen3 问题我已经解决，本轮不要关注。
     > 本轮只处理 `.wps` 和空 `.xls/.xlsx`。

  3. 验收标准

     写清楚什么叫“修好了”。

     示例：

     > `.wps` 能被识别为 Word 并进入解析。
     > 空 `.xls/.xlsx` 不应报“未解析到有效页面”或服务内部异常。
     > 解析结果可以为空页面/空元素，但任务不能失败。

  4. 提交范围

     如果需要提交 commit，提前要求清理无关改动。

     示例：

     > 提交前请 review diff，只提交与本 bug 直接相关的代码和测试。
     > debug 文档、复现脚本、其它格式修复不要进入 commit，除非我明确要求。

### 建议补充的内容

  - 样本路径和具体样本文件名。
  - 实际错误现象，例如截图、错误文案、任务状态、后端日志。
  - token / header 的用途和限制。
  - 是否需要从前端链路验证，而不是只跑后端单测。
  - 是否需要提交 commit，以及 commit 是否要拆分。
